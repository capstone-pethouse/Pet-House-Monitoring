# adr-002-20260505: 숫자 기반 PK/FK 전환 및 객체 지향적 연관관계 도입

## Status
- [x] Accepted

## Context
기존 시스템은 `Code`, `Device`, `Hospital` 등 주요 엔티티 간의 관계를 문자열(String) 기반의 비즈니스 키를 직접 저장하는 방식으로 관리하고 있었습니다. 이 방식은 다음과 같은 문제점이 있었습니다:
1. **성능 저하:** 문자열 기반의 JOIN은 정수형(BIGINT) 기반의 JOIN보다 DBMS 부하가 큼.
2. **무결성 부족:** 객체 참조가 아닌 단순 문자열 저장 방식으로 인해 JPA의 연관관계 매핑 이점을 활용하지 못하고, 참조 무결성 보장이 어려움.
3. **코드 중복:** 수동으로 ID를 조회하고 매핑하는 로직이 서비스 레이어 전반에 산재함.

## Decision
데이터베이스 성능 최적화와 객체 지향적 설계를 위해 다음과 같은 결정을 내렸습니다:
1. **PK 전환:** 모든 엔티티의 Primary Key를 `Long` 타입의 일련번호(`seq`)로 전환 (Identity 전략).
2. **연관관계 매핑:** 문자열 기반의 Soft FK를 JPA의 `@ManyToOne`, `@ManyToMany` 등 실제 객체 참조 방식으로 변경.
3. **비즈니스 키 유지:** 기존의 문자열 코드는 비즈니스 로직 및 외부 API 연동을 위해 `unique` 인덱스를 가진 컬럼으로 유지.
4. **브릿지 테이블 도입:** `Hospital`과 `Code`의 다대다 관계 해결을 위해 `@ManyToMany` 매핑과 중간 매핑 테이블(`hospital_med_code_map`)을 도입.

## Consequences
### 장점
- **성능 향상:** 숫자형 PK/FK 조인을 통한 쿼리 성능 최적화 및 인덱스 용량 감소.
- **개발 생산성:** JPA의 연관관계 편의 메서드와 지연 로딩(Lazy Loading)을 통한 객체 탐색 용이.
- **데이터 정합성:** DB 수준의 FK 제약 조건을 통한 엄격한 데이터 정합성 유지.

### 단점 및 트레이드오프
- **마이그레이션 복잡성:** 기존 데이터가 있는 경우, PK/FK 전환을 위한 별도의 DB 마이그레이션 작업이 필요함.
- **DTO 매핑 추가:** 엔티티 내부적으로는 `seq`를 사용하지만, API 스펙 유지를 위해 DTO 변환 시 문자열 코드로 변환하는 로직이 필요함.
